# Codebase Concerns

**Analysis Date:** 2026-03-25

## Tech Debt

| Area | Issue | Files | Impact | Fix Approach |
|------|-------|-------|--------|--------------|
| Uvicorn reload in production | `reload=True` is set in the uvicorn `Config` inside `_run()`. This flag is for development only — it spawns a file-watcher subprocess that adds overhead and can cause undefined behaviour inside Docker with bind mounts. | `src/main.py:56` | Unnecessary resource use; potential race conditions on startup inside containers | Set `reload=False`, or gate on an env var (`OA_DEV_RELOAD`) defaulting to `False` |
| WhatsApp inbound webhook has no authentication | `POST /webhook/whatsapp/baileys` accepts any JSON payload from any caller with no token, HMAC, or secret check. Only network isolation (bridge talks to `assistant` service via Docker internal DNS) currently protects this. | `src/channels/whatsapp.py:172-190` | Any process that can reach port 8080 can inject arbitrary messages into the agent | Add a shared secret between bridge and app (env var), validate it in the FastAPI handler |
| `gws-creds.json` path is doubly hardcoded | The bind mount in `docker-compose.yaml` mounts the file at the host path `/Users/beam/open-assitant/gws-creds.json` and maps it to the identical container path. This works only on the original developer's machine. Comment in compose explicitly flags "path hardcoded in gws config". | `docker-compose.yaml:28` | Deployment on any other machine requires manually editing `docker-compose.yaml` and the internal gws config | Use a named volume or env var path; normalise the container path to a predictable location like `/run/secrets/gws-creds.json` |
| `docker-compose.yaml` bind mounts hardcoded to `/Users/beam/` | Six bind mounts use absolute macOS paths (`/Users/beam/.open-assistant`, `/Users/beam/.claude/projects`, `/Users/beam/projects`, `/Users/beam/Downloads`, `/Users/beam/Documents`). | `docker-compose.yaml:20-32` | Cannot be deployed on any other machine or CI without manual edits | Replace with `${HOME}` or a project-level `.env` variable (e.g. `OPEN_ASSISTANT_DATA_DIR`) |
| Perplexity MCP server always registered | `_MCP_SERVERS` is built at module import time and always includes `perplexity-ask`, unconditionally passing `settings.perplexity_api_key` (which defaults to `""`). The MCP server process is started even when the key is absent, likely failing silently. | `src/agent/core.py:62-68` | Spurious subprocess startup on every agent `connect()`; confusing error logs when key is missing | Guard: only add the MCP config when `settings.perplexity_api_key` is non-empty |
| `_synthesize` (TTS) is dead code | `_synthesize()` is defined in `telegram.py` but never called anywhere after TTS voice replies were removed. | `src/channels/telegram.py:218-232` | Maintenance burden; OpenAI client is still imported and initialised regardless | Remove `_synthesize`, remove `openai` imports from the channel if not needed elsewhere; or leave with a comment if TTS is planned |
| Scheduler task errors are silently swallowed | `_run_task` has no top-level `try/except`. If `ask_agent` raises (e.g. network timeout, SDK error), APScheduler catches and logs at WARNING but the job silently fails with no user notification. | `src/scheduler/scheduler.py:48-71` | Scheduled briefings can fail invisibly | Wrap body in `try/except`, send a short error notification to the task's `notify` targets |
| Session store writes are not atomic | `save_session` does a bare `write_text()` with no temp-file + rename. A crash mid-write corrupts the session JSON, causing `load_session` to silently return `None` on next boot (losing resume capability). | `src/agent/session_store.py:33-34` | Session loss on unclean shutdown | Write to a `.tmp` file then `os.replace()` |

## Security Risks

| Area | Risk | Files | Mitigation | Recommendation |
|------|------|-------|------------|----------------|
| `.env` contains live API keys and is committed to source history risk | `.env` is in `.gitignore` but the file exists on disk with real credentials (`gws-creds.json` is also present). The `.dockerignore` does exclude `.env`, but `gws-creds.json` is explicitly bind-mounted into the container. | `.env`, `gws-creds.json`, `docker-compose.yaml:28` | Keys are not committed; `.dockerignore` excludes `.env` | Use Docker secrets or a secrets manager rather than bind-mounting credential files at their original filesystem path |
| Claude auth volume contains OAuth tokens with `Write` permissions | The `claude-auth` named volume is mounted at `/root/.claude`, which the agent has broad write access to (only `settings.json` is deny-listed). The volume persists `claude login` OAuth tokens. If the container is compromised, tokens can be exfiltrated. | `docker-compose.yaml:24`, `.claude/settings.json` | Named volume isolates from host; deny rule prevents settings overwrite | This is an acceptable trade-off for the use case, but the volume should never be `ro:false` on an internet-facing host without additional hardening |
| Inbound WhatsApp webhook is unauthenticated | `POST /webhook/whatsapp/baileys` at `src/channels/whatsapp.py:172` has no auth. Port 8080 is published to the host via `docker-compose.yaml:10`. If the host is internet-accessible, any HTTP client can inject messages into the agent. | `src/channels/whatsapp.py:172-190`, `docker-compose.yaml:10` | Currently relies on network isolation (port not forwarded via public reverse proxy) | Add a `BAILEYS_BRIDGE_SECRET` shared secret; validate an `X-Bridge-Token` header; or bind port 8080 to `127.0.0.1` only |
| Agent given broad Bash access including `grep -ril` over all Documents | `settings.json` grants `Bash(grep -ril * /root/Documents)` and similar patterns. This means the agent (or a prompt-injection attack via a message) can exfiltrate file contents from the bind-mounted host directories. | `.claude/settings.json` | Guardrail prompt says "confirm before acting"; confirmation requires user response | Tighten allowed Bash patterns; consider removing `-l` flag escalation to content grep |
| `_skill()` appends raw user input to SKILL.md content | In `telegram.py`, `_skill(name, args=args)` appends `f"\n\nUser input: {args}"` directly to the skill text that is sent as the agent prompt. A malicious user message that starts with instructions could attempt to override skill behaviour (prompt injection). | `src/channels/telegram.py:22-29` | User input is appended after the skill, not interpolated; guardrails in system prompt | Low risk but worth noting; consider wrapping user input in XML tags to make the boundary clearer |
| `baileys-bridge` Dockerfile uses `--mount=type=ssh` | The bridge image build mounts SSH keys at build time for private GitHub access. SSH agent state is build-time only and not baked into the image, but `npm install` uses the key, so the `node_modules` in the image layer reflects it. | `baileys-bridge/Dockerfile:3-8` | SSH key not stored in final image layer | Ensure `npm install` result is auditable; rotate the SSH key if the build cache is shared |

## Performance Bottlenecks

| Operation | Problem | Files | Cause | Fix |
|-----------|---------|-------|-------|-----|
| Memory pull on every startup | `memory_pull()` runs before the webhook server starts, performing multiple `gws` subprocess calls sequentially. If GDrive is slow or `gws` is not authenticated, startup is blocked for up to 30 s per GDrive call. | `src/main.py:34-39`, `src/memory/sync.py` | Synchronous startup gating | Run `memory_pull()` as a background task after the server is listening; the startup warning on failure already exists |
| New `httpx.AsyncClient` per Baileys bridge call | Every call to `_bridge_post` creates and tears down a new HTTPX client, including TLS handshake. The bridge is HTTP (not HTTPS) but the overhead is still unnecessary for high-frequency use. | `src/channels/whatsapp.py:33-42` | No shared client pool | Create a module-level `httpx.AsyncClient` with keepalive for the bridge, closed on app shutdown |
| One httpx client per scheduler Telegram notification | `send_telegram_message` opens a new `httpx.AsyncClient` for every 4 096-character chunk. On a long response this is multiple clients per notification. | `src/channels/telegram_notify.py:24-28` | Loop creates client inside loop body | Hoist client creation outside the loop |
| All GDrive operations run `gws` as a subprocess | Every file operation (list, download, upload) forks a new `gws` process. Syncing 10 memory files requires 20+ subprocesses. | `src/memory/sync.py:_run_gws` | `gws` CLI has no persistent daemon mode | Acceptable for current scale; if memory files grow, consider a Python GDrive SDK client instead |

## Fragile Areas

| Component | Files | Why Fragile | Safe Modification | Test Gaps |
|-----------|-------|-------------|-------------------|-----------|
| `entrypoint.sh` memory seeding | `entrypoint.sh:4-13` | Seeds `/root/.claude/projects` from `/initial-memory` only when the volume is empty. If the `claude-auth` volume is partially populated (e.g. after a failed first start), seeding is skipped and memory may be stale or missing. Logic uses `ls -A` which can be fooled by hidden files. | Always test with a fresh named volume after changing seeding logic | No automated test for entrypoint behaviour |
| `.claude.json` backup restore | `entrypoint.sh:22-28` | Restores Claude OAuth JSON from backup if missing. Depends on backups existing at `/root/.claude/backups/.claude.json.backup.*`. If the `claude-auth` volume is new and no backup exists, Claude CLI is unauthenticated and every agent call fails silently at connect time. | Run `docker exec -it assistant claude login` after first deploy | No test |
| `gws` CLI authentication | `src/memory/sync.py`, `docker-compose.yaml:23`, `.claude/skills/` | The `gws-auth` named volume holds OAuth state. If the volume is deleted or the token expires, all GDrive sync, email, calendar, and task operations silently fail. The agent will run but skill execution will produce errors that look like tool failures. | Implement `is_gdrive_available()` check at startup and log a clear actionable message | No test for expired-token behaviour |
| `_clients` dict in `core.py` is process-global | `src/agent/core.py:59` | `_clients` is a module-level dict. If uvicorn is in `reload=True` mode (current default), the reloader may restart the worker process, clearing `_clients` mid-conversation. Combined with the non-atomic session store write this can cause conversation context loss. | Disable `reload=True` (see Tech Debt) | No test for reload-induced state loss |
| `max_turns=15` hard cap | `src/agent/core.py:74` | If a complex skill (e.g. `week` briefing with many GDrive calls) requires more than 15 tool-call round-trips, the agent silently stops and returns a partial response with no indication to the user. | For agentic tasks likely to exceed 15 turns, increase limit or implement retry with continuation prompt | No test for max_turns cutoff |
| WhatsApp inbound only handles `text` type | `src/channels/whatsapp.py:184-188` | Messages of type `voice`, `image`, `document`, etc. are silently ignored with `{"status": "ignored"}`. There is no fallback to inform the sender that their message was not processed. | If non-text handling is added, ensure the `detectMessageType` logic in the bridge matches exactly | No test for ignored message types |
| Scheduler tasks share a `sched:{name}` chat_id | `src/scheduler/scheduler.py:57` | All runs of a scheduled task share a single persistent Claude session keyed on the task name. History accumulates indefinitely. On long-running deployments the context window will fill up and trigger auto-compact, which may silently truncate task context. | Add a session reset mechanism for scheduled tasks (e.g. reset every N runs) | No test for context accumulation |
| Telegram notification uses plain text, not MarkdownV2 | `src/channels/telegram_notify.py:27` | Scheduled notifications are sent as plain text without `markdownify` conversion, while interactive responses use MarkdownV2. Markdown symbols in agent output (e.g. `**bold**`) will appear as raw characters in scheduled messages. | Apply `markdownify` and `parse_mode="MarkdownV2"` in `send_telegram_message` | No test |

## Dependencies at Risk

| Package | Risk | Impact | Migration Plan |
|---------|------|--------|----------------|
| `claude-agent-sdk>=0.1.48` | Internal Anthropic SDK, alpha-stage. API surface has changed across minor versions (e.g. `SystemMessage` session_id field). Breaking changes require immediate code updates. | All agent functionality | Pin to an exact version in `requirements.txt`; test each upgrade before deploying |
| `@googleworkspace/cli` (npm, global) | Installed globally in the `Dockerfile` with no version pin (`npm install -g @googleworkspace/cli`). A breaking CLI update silently breaks all GDrive sync and GWS skills on the next image rebuild. | All GDrive/email/calendar features | Pin the version (`@googleworkspace/cli@<version>`) in the `Dockerfile` |
| `@anthropic-ai/claude-code` (npm, global) | Also installed globally without a version pin. The `claude` CLI is used for OAuth login and session management. A breaking change affects all authentication. | Claude auth, `claude login` workflow | Pin the version in the `Dockerfile` |
| `@whiskeysockets/baileys` | WhatsApp Web reverse-engineered protocol. WhatsApp regularly changes the protocol; Baileys releases breaking updates to stay compatible. No version is pinned in `baileys-bridge/package.json` (uses `npm install` which resolves `^` ranges). | All WhatsApp functionality | Pin to a specific version; monitor Baileys releases actively |
| `python-telegram-bot>=21.0` | Broad minor-version range. PTB v21 changed the async application lifecycle significantly vs v20. Future minor versions may introduce subtle incompatibilities. | Telegram channel | Pin minor version (e.g. `>=21.0,<22.0`) |
